# Specs-Driven Development (SDD)

> The skill is no longer coding — it's requirements definition.

## Core Idea

SDD treats software as a compiled artifact:

Specification → Code (generated by AI agents)

Like Terraform, SQL, or hardware description languages — but applied to entire applications.

The human role shifts from writing code to defining precise specifications.

## The 3 Principles

### 1. Specs replace code as the core skill

AI generates most code. Human value shifts to defining requirements and system design with precision.

| Old Model | New Model |
|-----------|-----------|
| Dev writes code | AI writes code |
| PM defines rough requirements | Technical people define precise specs |
| Tech lead reviews code | Tech lead reviews design & specs |

### 2. Fix specs, not code

When a bug occurs, you don't patch the implementation — you fix the specification and regenerate.

- ❌ Bug → patch code
- ✅ Bug → fix spec → regenerate code

### 3. Tests are part of the spec layer

AI generates unit tests and UI tests from specs. Engineers focus on testing methodology, not manual test-writing.

- Spec → generate code
- Spec → generate tests
- Tests validate the system

## Workflow

1. Write structured specs (`proposal.md` → `design.md` → `tasks.md`)
2. Human review of design (not code)
3. Feed spec to coding agent → implementation
4. Generate tests (AI-driven)
5. Execute

If bug: fix spec → regenerate (not hotfix).

Key principle: The agent generates `proposal.md` first and waits for human review before proceeding.

## Tooling Direction

- Spec files: `proposal.md`, `design.md`, `tasks.md`
- Agent config: `AGENTS.md` installed per-project like a package version
- Tools discussed: OpenSpec, Claude Code, coding agents

Flow:

Central skill registry → project imports agent config (`AGENTS.md`) → agent uses context → generates code artifacts

## Team Structure (Projected)

Architect / System Designer
↓
Specification Authors
↓
Coding Agents
↓
Junior Devs validating outputs

Senior engineers mainly define specs, review designs, and arbitrate technical decisions.

## Why Now?

SDD was attempted several times since the 1990s but failed.
The current viability (2025–2026) comes from:

- stronger AI coding agents
- structured LLM outputs
- better agent orchestration tooling

## Open Questions

- How do specs handle emergent complexity not anticipated upfront?
- Who owns the specs when teams are small?
- How should specs be versioned alongside generated code?
- What happens when generated output diverges from spec intent?
